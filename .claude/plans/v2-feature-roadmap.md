# Docufinder (Anything) 전략 분석 보고서
## 세계적 파일검색 RAG + 공무원 특화 기능 로드맵

---

## 1. 현재 프로젝트 진단

### 1.1 아키텍처 평가 (30년차 시니어 관점)

| 영역 | 평가 | 점수 |
|------|------|------|
| **Clean Architecture** | 도메인/응용/인프라 분리 우수. Repository 패턴 적용 | ⭐⭐⭐⭐ |
| **검색 엔진** | FTS5 + usearch HNSW + RRF 하이브리드. 학술 수준 | ⭐⭐⭐⭐⭐ |
| **동시성 모델** | 락 순서 고정, Atomic Save, TOCTOU 방어 완비 | ⭐⭐⭐⭐ |
| **보안** | Zip Bomb 방어, CSP, SHA-256 검증, 경로순회 차단 | ⭐⭐⭐⭐ |
| **파서 견고성** | catch_unwind, 타임아웃, 메모리 제한 | ⭐⭐⭐⭐ |
| **UX 완성도** | IME 처리, 키보드 내비게이션, 접근성 | ⭐⭐⭐⭐ |
| **성능 최적화** | 2단계 인덱싱, SSD/HDD 감지, mmap View 모드 | ⭐⭐⭐⭐⭐ |

**종합: 8.5/10** — 로컬 검색 도구로서 아키텍처적으로 매우 잘 설계됨. 상용 제품 수준.

### 1.2 현재 강점 (경쟁 우위)

1. **완전 오프라인**: 클라우드 의존 제로 → 공공기관 보안 요구 충족
2. **한국어 특화**: KoSimCSE + Lindera 형태소 분석 → 한국어 시맨틱 검색
3. **HWPX 네이티브 지원**: 한글 문서 직접 파싱 (경쟁자 대비 차별점)
4. **하이브리드 검색**: 키워드 + 시맨틱 + RRF + Cross-Encoder 재정렬
5. **경량**: Tauri (Electron 대비 10x 작은 번들)

### 1.3 현재 약점/한계

| 약점 | 영향도 | 개선 난이도 |
|------|--------|------------|
| Windows 전용 | 시장 제한 | 높음 (macOS/Linux 포팅) |
| 단일 사용자 | 팀 협업 불가 | 높음 (서버 아키텍처 필요) |
| 파일 미리보기 없음 | UX 불편 | 중간 |
| 검색어 추천 없음 | 검색 효율 저하 | 중간 |
| 문서 분류/태깅 없음 | 정리 불가 | 중간 |
| 다국어 미지원 | 영문 검색 품질 낮음 | 낮음 |
| PPT/PPTX 미지원 | 발표자료 검색 불가 | 낮음 |

## 2. 공무원 특화 기능 제안 (우선순위순)

### Tier 1: 킬러 기능 (즉시 차별화)

---

#### 2.1 📋 유사 공문서 찾기 (Similar Document Finder)

**문제**: 공무원이 새 공문서 작성 시, 과거 유사 문서를 찾아 참조하는 데 평균 30-60분 소요

**기능**:
- 현재 작성 중인 문서를 드래그&드롭 → 유사도 상위 N개 문서 반환
- 클립보드 텍스트 붙여넣기 → 유사 문서 검색
- 검색 결과에서 "이 문서와 유사한 문서 찾기" 버튼

**구현 방식**:
- 이미 보유한 벡터 인덱스 + KoSimCSE 임베딩 활용
- 입력 문서를 청킹 → 각 청크 벡터 검색 → 파일 단위 집계 → 유사도 순 정렬
- 기존 `search_service.rs`의 시맨틱 검색 파이프라인 재활용

**실현가능성**: ⭐⭐⭐⭐⭐ (핵심 인프라 이미 존재)
**유용성**: ⭐⭐⭐⭐⭐ (공무원 #1 페인포인트)
**예상 공수**: 2-3일 (프론트 UI + 백엔드 엔드포인트)

---

#### 2.2 🏷️ 자동 문서 분류 (Auto Document Classification)

**문제**: 수천 개 문서가 폴더별로만 정리, 내용 기반 분류 없음

**기능**:
- 인덱싱 시 문서를 자동으로 카테고리 분류:
  - `공문`, `보고서`, `회의록`, `기안문`, `시행문`, `계획서`, `결과보고`, `법령/규정`, `통계`, `기타`
- 검색 필터에 "문서 유형" 드롭다운 추가
- 문서 목록에 유형 뱃지 표시

**구현 방식**:
- **방법 A (규칙 기반)**: 제목/첫 문단 키워드 패턴 매칭 (정확도 ~75%)
  - "시행", "기안", "결재" → 공문
  - "회의록", "참석자" → 회의록
  - "제N조", "법률", "시행령" → 법령
- **방법 B (임베딩 기반)**: 카테고리별 앵커 텍스트 정의 → 코사인 유사도로 분류 (정확도 ~85%)
  - 기존 KoSimCSE 모델 재활용, 추가 모델 불필요
- **추천**: B 방식 (이미 임베딩 인프라 있으므로 추가 비용 최소)

**실현가능성**: ⭐⭐⭐⭐ (임베딩 인프라 활용)
**유용성**: ⭐⭐⭐⭐⭐ (감사 대비, 문서 정리)
**예상 공수**: 3-5일

---

#### 2.3 📄 문서 미리보기 패널 (Document Preview Pane)

**문제**: 검색 결과 클릭 → 외부 앱(한글, Word)으로 열림 → 확인 후 닫기 반복

**기능**:
- 검색 결과 우측에 미리보기 패널 (VS Code 스타일)
- 텍스트 콘텐츠 표시 + 검색어 하이라이트
- 해당 청크 위치로 자동 스크롤
- 페이지 정보 표시 (PDF 페이지, XLSX 시트)

**구현 방식**:
- 이미 파싱된 텍스트 콘텐츠를 DB에서 로드 (chunks 테이블)
- 파일별 전체 청크를 순서대로 조합 → 텍스트 렌더링
- 검색어 위치에 하이라이트 마커 표시
- 우측 패널 (resizable, 토글 가능)

**실현가능성**: ⭐⭐⭐⭐⭐ (데이터 이미 DB에 존재)
**유용성**: ⭐⭐⭐⭐⭐ (검색 효율 2-3배 향상)
**예상 공수**: 3-4일

### Tier 2: 생산성 부스터

---

#### 2.4 🔍 검색어 자동완성/추천 (Query Suggestion)

**문제**: 공무원이 정확한 검색어를 모를 때 여러 번 시도

**기능**:
- 타이핑 시 자동완성 드롭다운 (최근 검색 + 문서 내 빈출 용어)
- "이 검색어로도 시도해보세요" 추천 (시맨틱 유사 용어)
- 오타 교정 ("보고소" → "보고서")

**구현 방식**:
- FTS5의 `fts5vocab` 가상 테이블 → 인덱싱된 용어 빈도 추출
- 최근 검색 기록 기반 prefix matching
- 시맨틱 유사어: 입력 쿼리 임베딩 → 상위 N개 유사 청크 → 핵심 용어 TF-IDF 추출

**실현가능성**: ⭐⭐⭐⭐ (FTS5 vocab 테이블 활용)
**유용성**: ⭐⭐⭐⭐ (검색 시행착오 감소)
**예상 공수**: 3-4일

#### 2.5 📊 문서 통계 대시보드 (Document Analytics)

**문제**: "내 PC에 어떤 문서가 얼마나 있는지" 파악 불가

**기능**:
- 대시보드 뷰:
  - 파일 유형별 분포 (도넛 차트)
  - 연도별 문서 생성 추이 (라인 차트)
  - 폴더별 문서 수 (트리맵)
  - 최근 수정된 문서 Top 10
  - 가장 큰 문서 Top 10
  - 인덱싱 현황 (전체/완료/실패)
- 각 통계 클릭 → 해당 문서 필터링된 검색 결과로 이동

**구현 방식**:
- SQLite 집계 쿼리 (`GROUP BY type`, `strftime('%Y', modified_at)`)
- 프론트엔드: 순수 CSS/SVG 차트 (추가 라이브러리 최소화)
- 또는 경량 차트 라이브러리 (recharts ~45KB gzipped)

**실현가능성**: ⭐⭐⭐⭐⭐ (DB 쿼리만으로 구현 가능)
**유용성**: ⭐⭐⭐⭐ (현황 파악, 감사 대비)
**예상 공수**: 3-4일

#### 2.6 📌 문서 북마크 + 메모 (Bookmarks & Notes)

**문제**: 중요 문서를 발견해도 나중에 다시 찾기 어려움

**기능**:
- 검색 결과에서 ⭐ 북마크 토글
- 북마크에 간단한 메모 추가 ("2024년 감사 대비 자료", "이 양식 사용할 것")
- 사이드바에 북마크 폴더 (태그별 그룹핑)
- 북마크 내 검색

**구현 방식**:
- SQLite 테이블 추가: `bookmarks (id, file_path, note, tags, created_at)`
- 프론트엔드: 사이드바 탭 추가, 북마크 카드 컴포넌트

**실현가능성**: ⭐⭐⭐⭐⭐ (단순 CRUD)
**유용성**: ⭐⭐⭐⭐ (개인 지식 관리)
**예상 공수**: 2-3일

#### 2.7 📁 PPTX 파서 추가 (PowerPoint Support)

**문제**: 공무원은 발표자료/보고서를 PPTX로 많이 작성. 현재 검색 불가

**기능**:
- `.pptx` 파일 인덱싱 + 검색
- 슬라이드 번호별 위치 정보
- 노트 영역 텍스트도 인덱싱

**구현 방식**:
- PPTX는 ZIP + XML 구조 (DOCX와 동일 패턴)
- `ppt/slides/slide{N}.xml` 파싱
- 기존 `validate_zip_archive()` 보안 로직 재활용

**실현가능성**: ⭐⭐⭐⭐⭐ (DOCX 파서와 거의 동일 패턴)
**유용성**: ⭐⭐⭐⭐ (발표자료 검색)
**예상 공수**: 2일

#### 2.8 🤖 AI 문서 요약 (Document Summarization)

**문제**: 긴 보고서의 핵심 내용을 빠르게 파악하고 싶음

**기능**:
- 검색 결과에서 "요약 보기" 버튼
- 문서 전체 텍스트 → 3-5문장 요약
- 미리보기 패널 상단에 요약 표시

**구현 방식**:
- **방법 A (추출적 요약)**: TextRank 알고리즘 (Rust 구현, 오프라인)
  - 문장별 TF-IDF 벡터화 → 문장 간 유사도 그래프 → PageRank → 상위 N 문장
  - 추가 모델 불필요, 100% 오프라인
  - 품질: 중상 (핵심 문장 추출이므로 자연스러움)
- **방법 B (생성적 요약)**: 로컬 LLM (e.g., Phi-3-mini ONNX)
  - 추가 ~2GB 모델 다운로드 필요
  - 요약 품질 우수하지만 리소스 부담

**추천**: A 방식 우선 (오프라인, 경량), 향후 B 옵션 추가

**실현가능성**: ⭐⭐⭐⭐ (A 방식, 알고리즘 구현)
**유용성**: ⭐⭐⭐⭐⭐ (보고서 검토 시간 대폭 절감)
**예상 공수**: 4-5일 (A 방식)

#### 2.9 🔗 문서 관계 맵 (Document Relationship Graph)

**문제**: 관련 문서들이 여러 폴더에 흩어져 있어 전체 맥락 파악 어려움

**기능**:
- 선택한 문서와 유사도 높은 문서들을 시각적 그래프로 표시
- 노드 = 문서, 엣지 = 유사도 (두께로 표현)
- 클러스터 자동 감지 (같은 주제 문서군)
- 노드 클릭 → 문서 상세 보기

**구현 방식**:
- 벡터 인덱스의 KNN 검색 활용 (이미 구현됨)
- 프론트엔드: D3.js force-directed graph 또는 Canvas 기반
- 문서 단위 평균 임베딩 → 유사도 매트릭스 → 그래프 렌더링

**실현가능성**: ⭐⭐⭐ (벡터 인프라 있지만 문서 단위 임베딩 필요)
**유용성**: ⭐⭐⭐⭐ (지식 관리, 업무 인수인계)
**예상 공수**: 5-7일

#### 2.10 📝 OCR 지원 (스캔 문서 검색)

**문제**: 공무원이 스캔한 PDF, 팩스 수신 문서가 많음 → 텍스트 추출 불가

**기능**:
- 이미지 기반 PDF 감지 (텍스트 레이어 없는 PDF)
- OCR 엔진으로 텍스트 추출 → 인덱싱
- 스캔 이미지 파일 (jpg, png, tiff)도 지원

**구현 방식**:
- **Tesseract OCR** (오픈소스): `leptess` Rust 바인딩
- 한국어 학습 데이터 (`kor.traineddata` ~15MB) 번들링
- 파서 파이프라인에 OCR 분기 추가

**실현가능성**: ⭐⭐⭐ (외부 의존성 + 번들 크기 증가 ~50MB)
**유용성**: ⭐⭐⭐⭐⭐ (스캔 문서 검색 = 공무원 핵심 니즈)
**예상 공수**: 5-7일

#### 2.11 🏛️ 법령/규정 참조 링크 (Legal Reference Linker)

**문제**: 문서 내 "제15조", "시행령 제3조" 등 법령 참조가 있지만 원문 확인이 번거로움

**기능**:
- 문서 미리보기에서 법령 참조 자동 감지 (정규식)
- "제N조", "법률 제NNNNN호", "시행령", "시행규칙" 패턴 하이라이트
- 클릭 → 법제처 국가법령정보센터 링크 (law.go.kr)
- 오프라인 시 참조 목록만 표시

**구현 방식**:
- 정규식 기반 법령 참조 추출: `제\d+조`, `법률\s*제\d+호`
- URL 템플릿: `https://law.go.kr/검색?query={법령명}`
- 프론트엔드: 미리보기 패널에 인터랙티브 링크 렌더링

**실현가능성**: ⭐⭐⭐⭐ (정규식 + URL 매핑)
**유용성**: ⭐⭐⭐⭐ (법무/행정 공무원 필수)
**예상 공수**: 2-3일

#### 2.12 📤 검색 결과 내보내기 강화 (Export Enhancement)

**문제**: 검색 결과를 보고서/메일에 첨부해야 할 때 복사가 번거로움

**기능**:
- 검색 결과 → Excel 내보내기 (파일명, 경로, 유사도, 매칭 텍스트, 날짜)
- 검색 결과 → PDF 보고서 생성 ("문서 검색 결과 보고서")
- 선택한 문서들 → ZIP 패키징 (문서 모아서 전달)
- 결과 공유 링크 (로컬 서버 기반)

**구현 방식**:
- CSV export는 이미 `useExport` 훅에 존재 → XLSX 확장
- PDF 보고서: `printpdf` Rust crate 또는 HTML→PDF
- ZIP 패키징: `zip` crate (이미 의존성에 포함)

**실현가능성**: ⭐⭐⭐⭐ (기존 인프라 활용)
**유용성**: ⭐⭐⭐⭐ (보고서 작성, 감사 대응)
**예상 공수**: 3-4일

#### 2.13 🗂️ 중복 문서 탐지 (Duplicate Document Detection)

**문제**: 같은 문서가 여러 폴더에 복사되어 저장공간 낭비 + 버전 혼란

**기능**:
- "중복 문서 찾기" 전용 탭/모드
- 정확한 중복 (해시 일치) + 유사 중복 (내용 90%+ 유사)
- 중복 그룹별 표시 + 원본/사본 추천
- 일괄 삭제/이동 기능

**구현 방식**:
- **정확 중복**: files 테이블의 `hash` 컬럼 활용 (이미 존재)
- **유사 중복**: 문서 단위 평균 벡터 → 코사인 유사도 0.95+ 필터
- 프론트엔드: 그룹 카드 UI

**실현가능성**: ⭐⭐⭐⭐ (해시 이미 저장됨)
**유용성**: ⭐⭐⭐⭐ (저장공간 관리, 문서 정리)
**예상 공수**: 3-4일

#### 2.14 ⏰ 문서 만료 알림 (Document Expiry Alerts)

**문제**: 계약서, 인허가, MOU 등 유효기간 있는 문서 관리 누락

**기능**:
- 문서 내 날짜 패턴 자동 감지 ("2025년 3월 31일까지", "유효기간: ~2025.12.31")
- 만료 임박 문서 알림 (30일/7일/당일)
- 캘린더 뷰 (문서 만료 타임라인)
- Windows 시스템 알림 연동

**구현 방식**:
- 정규식 기반 날짜 추출: `\d{4}[년./-]\s*\d{1,2}[월./-]\s*\d{1,2}[일]?`
- "만료", "유효기간", "까지", "종료" 키워드 근접 날짜를 만료일로 추정
- SQLite 테이블: `document_dates (file_id, date_type, date_value, context)`
- Tauri 시스템 알림: `tauri-plugin-notification`

**실현가능성**: ⭐⭐⭐ (날짜 추출 정확도가 도전 과제)
**유용성**: ⭐⭐⭐⭐⭐ (계약/인허가 관리 = 공무원 핵심 업무)
**예상 공수**: 5-7일

## 4. 세계적 수준을 위한 아키텍처 제언

### 4.1 현재 vs 목표 비교

| 영역 | 현재 | 세계적 수준 (Elasticsearch/Algolia급) |
|------|------|--------------------------------------|
| 검색 품질 | RRF 하이브리드 | ✅ 이미 수준급 |
| 다국어 | 한국어만 | 다국어 토크나이저 + 언어별 임베딩 |
| 스케일 | ~100만 문서 | 1000만+ (샤딩, 분산) |
| 실시간성 | 2초 폴링 | <100ms (inotify/ReadDirectoryChanges) |
| 검색 분석 | 없음 | 쿼리 로그, CTR, 검색 품질 메트릭 |
| 개인화 | 없음 | 사용자 행동 기반 결과 부스팅 |
| 피드백 루프 | 없음 | 검색 결과 피드백 → 랭킹 개선 |

### 4.2 단기 아키텍처 개선 (코드 변경 최소)

1. **검색 품질 피드백**: 결과 클릭 로그 → 클릭한 결과 부스팅 (SQLite 테이블)
2. **쿼리 확장**: 시맨틱 유사어 자동 추가 (기존 임베딩 활용)
3. **적응형 RRF k값**: 쿼리 유형별 k 조정 (짧은 쿼리 vs 긴 쿼리)
4. **결과 다양성**: MMR (Maximal Marginal Relevance) 적용하여 중복 결과 억제

### 4.3 중기 아키텍처 확장

1. **플러그인 시스템**: 파서를 플러그인 아키텍처로 전환 (DLL/WASM)
2. **인덱스 샤딩**: 폴더 단위 벡터 인덱스 분할 (메모리 효율)
3. **증분 벡터 업데이트**: 변경된 청크만 재임베딩 (현재는 파일 단위)
4. **멀티모달**: 이미지 임베딩 (CLIP) → 이미지 내용 검색

## 5. 공무원 시장 진입 전략

### 5.1 공무원이 겪는 문서 관련 페인포인트

| 순위 | 문제 | 현재 해결 방법 | Anything 해결 |
|------|------|---------------|---------------|
| 1 | 과거 유사 공문 찾기 | 폴더 뒤지기, 동료에게 물어봄 | ① 유사문서 검색 |
| 2 | 스캔 문서에서 내용 찾기 | 눈으로 읽거나 포기 | ⑩ OCR |
| 3 | 계약/인허가 만료 관리 | Excel 대장 수기 관리 | ⑭ 만료알림 |
| 4 | 감사 대비 자료 정리 | 수작업 폴더 정리 | ② 분류 + ⑤ 통계 |
| 5 | 법령 참조 확인 | 법제처 수동 검색 | ⑪ 법령링크 |
| 6 | 문서 버전 혼란 | 파일명에 날짜 추가 | ⑬ 중복탐지 |
| 7 | 업무 인수인계 | 폴더구조 설명 문서 | ⑨ 관계맵 |

### 5.2 도입 시나리오

**Phase A**: 개인 사용 (현재) → 공무원 개인 PC에 설치
- 완전 오프라인, 개인정보 유출 위험 없음
- 별도 서버/네트워크 구성 불필요
- 관리자 권한 불필요 (MSI 설치만)

**Phase B**: 부서 공유 → 공유 폴더 인덱싱
- NAS/파일서버의 공유 폴더를 감시 대상 추가
- 각 PC에서 개별 인덱싱 (중앙 서버 없음)

**Phase C**: 조직 전체 → 서버 배포 (v4.0)
- 웹 버전으로 조직 전체 문서 검색
- 권한 관리, 감사 로그 추가

### 5.3 공공기관 보안 요구사항 충족

| 요구사항 | 충족 여부 | 비고 |
|----------|----------|------|
| 망분리 환경 운영 | ✅ | 완전 오프라인 |
| 개인정보 외부 전송 금지 | ✅ | 네트워크 통신 없음 |
| 국정원 보안 가이드라인 | ✅ | 로컬 전용, 암호화 가능 |
| CC인증 | ⚠️ | 필요 시 별도 인증 절차 |
| 소프트웨어 목록 등록 | ⚠️ | 조달청 나라장터 등록 가능 |

## 6. 결론

### 현재 Anything의 위치
- **기술적 완성도**: 상용 제품 수준 (8.5/10)
- **검색 품질**: 학술 논문급 하이브리드 아키텍처
- **차별점**: 한국어 + HWPX + 완전 오프라인 = 경쟁자 없는 블루오션

### 세계적 수준까지의 거리
- **검색 엔진**: 이미 80% 수준 (RRF + Cross-Encoder + 한국어 시맨틱)
- **UX**: 60% (미리보기, 분류, 추천 부재)
- **공무원 특화**: 30% (핵심 기능 미구현)

### 최우선 투자 영역
1. **미리보기 패널** — 즉각적 체감 개선, 낮은 비용
2. **유사 문서 찾기** — 킬러 기능, 기존 인프라 100% 활용
3. **자동 분류** — 공무원 차별화, 감사 대비
4. **OCR** — 스캔 문서 = 공공기관 최대 사각지대

> **핵심 메시지**: Anything은 이미 견고한 검색 엔진을 보유하고 있으며,
> 위 기능들은 기존 인프라(벡터 인덱스, 임베딩, 파서 파이프라인)를
> **재활용**하여 구현 가능하므로, 투자 대비 효과가 극대화됨.
