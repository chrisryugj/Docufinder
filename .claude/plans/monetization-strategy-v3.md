# 월 30만원 수익화 전략 (v3 - 법적 현실 반영)

## Context
공무원 사이드 프로젝트로 월 30만원 구독 수익 목표.
IP 본인 소유, 부모 법인(SW 업종 추가 가능), 부모님 실질 운영 참여 어려움.
주 5-10시간 마케팅 가능.

---

## Part 1: 법적 검토 결과

### 법제처 API 상업적 이용 — OK
- 공공데이터포털: 공공누리 **제1유형** (출처표시)
- **상업적 이용 가능**, 2차 저작물 가능
- 트래픽: 개발 10,000건/일, 운영계정은 신청 시 증가
- 출처: "법제처 국가법령정보 Open API" 표기만 하면 됨

### 공무원 겸직 제한 — 핵심 리스크
- 국가공무원법 제64조: **영리업무 종사 금지**
- 복무규정 제25조: 영리적 업무 경영, 타인 기업 이사/감사 금지
- 판단 기준: **명의 무관, "실질적 종사 여부"로 판단**
- 부모님 실질 운영 어려움 → 본인이 사실상 운영 = **부모 법인 우회가 법적으로 위험**

### 현실적 판단

부모님이 결제/CS/마케팅을 실질적으로 못하면, 부모 법인은 **형식적 명의 대여**에 불과.
이 경우 "실질적 종사"로 판단될 리스크가 있음.

→ **3가지 경로를 리스크 순으로 제시**

## Part 2: 3가지 경로 (리스크 순)

### 경로 A: 겸직허가 → 정식 상용화 (가장 안전)

**개요:** 소속기관에 겸직허가를 신청하고, 승인 후 부모 법인으로 상용 서비스 운영

**근거:**
- 최근 유튜브/블로그/인터넷방송 수익 겸직허가 완화 추세
- SW 개발/판매도 같은 "지식 콘텐츠 창작" 논리 적용 가능
- 소속기관장이 개별 판단 (담당 직무수행 지장 여부)

**절차:**
1. 겸직허가 신청서 작성 (업무 내용: 소프트웨어 개발 및 판매)
2. 소속기관장 심사 (직무 지장, 이해충돌 검토)
3. 허가 시 → 부모 법인으로 결제/세금 처리
4. 본인은 법인 직원이 아닌 "외부 개발자"로 용역 제공

**장점:** 완전 합법, 투명, 장기적 안정
**단점:** 허가가 안 날 수 있음, 신청 자체가 부담

**소요 시간:** 1-2개월 (신청~결과)

---

### 경로 B: 오픈소스 + 후원/기부 모델 (중간 리스크)

**개요:** 모든 제품을 무료 오픈소스로 유지하되, 후원금을 받는 모델

**구조:**
```
[오픈소스 프로젝트] → GitHub Sponsors / Buy Me a Coffee / Open Collective
                    → "기부"로 수령 (대가 없는 후원)
```

**왜 상대적으로 안전한가:**
- "영리업무 종사"는 계속적 재산상 이득을 위한 **업무**
- 오픈소스 기여 자체는 비영리 활동
- 후원은 "대가 없는 기부"로 볼 여지
- 유튜브 슈퍼챗과 유사한 성격
- 다만 엄밀히는 "계속적 재산상 이득"에 해당할 수 있어 겸직허가 권장

**수익 극대화 방법:**
1. GitHub Sponsors 프로필 최적화
2. README에 Sponsor 버튼 + "후원하기" 배지
3. 후원자 전용 Discord 채널 (기술 Q&A)
4. 후원자 전용 기능 (early access, priority issue)

**현실적 수익 예측:**
- GitHub Sponsors: 매우 어려움 (한국 개발자 Sponsors 생태계 미성숙)
- Buy Me a Coffee: 월 5-10만원 수준이 현실적
- **월 30만원은 후원만으로 달성 어렵다** → 보조 수단

**장점:** 리스크 낮음, 즉시 시작 가능
**단점:** 월 30만원 달성 현실적으로 어려움

---

### 경로 C: 부모 법인 + 최대 자동화 (가장 높은 리스크, 가장 현실적)

**개요:** 부모 법인 명의로 상용 서비스 운영, 모든 것을 자동화하여 부모 개입 최소화

**자동화로 부모 부담 줄이기:**
| 업무 | 자동화 방법 | 부모 역할 |
|------|-------------|----------|
| 결제 | Stripe 자동 처리 | 없음 (자동) |
| 키 발급 | 결제 완료 시 자동 발급 | 없음 (자동) |
| CS | FAQ 페이지 + 자동 이메일 | 없음 (셀프서비스) |
| 세금 | 세무사 위임 (분기별) | 세무사 계약 서명 |
| 통장 | 법인 통장 → 자동 관리 | 분기별 확인 |

**부모님이 실질적으로 해야 할 것:**
- 법인 사업 목적 추가 등기 서명 (1회)
- 세무사 계약 서명 (1회)
- Stripe 법인 계정 개설 시 서명 (1회)
- 분기별 세무 확인 서명 (연 4회)

→ 실질적으로 **연 5-6회 서명** 수준

**리스크:**
- 본인이 실질적 운영자임이 밝혀지면 "영리업무 종사"로 판단 가능
- 현실: 월 30만원 규모의 소규모 SaaS가 적발될 확률은 극히 낮음
- 하지만 법적으로는 위반에 해당할 수 있음

**리스크 완화:**
- 오픈소스(MIT) + 상용 분리 구조 유지
- 법인 명의 = 모든 결제/계약/세금
- 본인 명의 = GitHub 오픈소스 기여만
- 법인 홈페이지에 본인 이름 노출 안 함

## Part 3: 추천 전략 — 경로 A+C 하이브리드

### Phase 1 (즉시~1개월): 경로 B로 시작 + 겸직허가 준비

**즉시 실행:**
1. korean-law-mcp, Docufinder에 GitHub Sponsors / Buy Me a Coffee 설정
2. README에 후원 배지 추가
3. 블로그/커뮤니티 마케팅 시작 (무료 제품 홍보)

**병행:**
4. 겸직허가 신청 검토 (인사팀 비공식 문의)
5. 부모 법인 사업 목적 추가 등기 준비

### Phase 2 (1~2개월): 겸직허가 결과에 따라 분기

**허가 OK → 경로 A (정식 상용화)**
- 부모 법인으로 상용 서비스 출시
- 본인이 공개적으로 개발자/운영자
- 완전 합법, 투명

**허가 거절 or 신청 포기 → 경로 C (자동화 + 법인)**
- 부모 법인 명의 상용 서비스
- 최대 자동화
- 리스크 인지하고 진행

### Phase 3 (2~4개월): 제품 출시

## Part 6: 검증 방법

1. **수요 검증** (제품 출시 전)
   - 랜딩 페이지만 먼저 배포 → 이메일 수집
   - 이메일 100개 모이면 → 개발 착수
   - 안 모이면 → 가치 제안/타겟 재검토

2. **전환 검증** (출시 후 1개월)
   - 무료 사용자 50명 중 유료 전환 1명 이상 → 계속
   - 전환 0명 → 가격/기능 실험 or 피봇

3. **지속 검증** (월간)
   - MRR, Churn rate, CAC 트래킹
   - Stripe Dashboard로 자동 확인
