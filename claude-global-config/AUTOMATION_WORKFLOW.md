# 자동화 워크플로우 설계

> 모든 컴포넌트가 유기적으로 맞물려 자동으로 동작하는 시스템

---

## 1. 자동화 철학

### 1.1 핵심 원칙

```
"사용자가 외워야 할 것은 없다"
"시스템이 알아서 트리거하고 안내한다"
"실수를 사전에 방지한다"
```

### 1.2 자동화 레이어

```
┌─────────────────────────────────────────────────────────────┐
│                    Layer 4: 세션 라이프사이클                  │
│         SessionStart → 작업 → SessionEnd → Handoff          │
├─────────────────────────────────────────────────────────────┤
│                    Layer 3: 이벤트 트리거                      │
│     Hooks (Pre/Post ToolUse, Compaction, Stop)              │
├─────────────────────────────────────────────────────────────┤
│                    Layer 2: 워크플로우 자동화                  │
│     Commands → Agents 자동 위임 → 검증 루프                   │
├─────────────────────────────────────────────────────────────┤
│                    Layer 1: 기본 규칙                        │
│     Rules (Security, Style, Testing, Git) 상시 적용          │
└─────────────────────────────────────────────────────────────┘
```

## 2. 자동 트리거 시스템

### 2.1 세션 시작 자동화

```
┌─────────────────┐
│  세션 시작       │
└────────┬────────┘
         │
         ▼
┌─────────────────────────────────────┐
│  SessionStart Hook 자동 실행         │
│  ┌─────────────────────────────────┐│
│  │ 1. HANDOFF.md 존재 확인          ││
│  │ 2. 있으면 → 자동 읽기 제안        ││
│  │ 3. 이전 세션 컨텍스트 복원        ││
│  │ 4. 패키지 매니저 자동 감지        ││
│  └─────────────────────────────────┘│
└─────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────┐
│  "이전 세션 작업을 이어서 할까요?"    │
│  📋 HANDOFF.md 발견됨               │
└─────────────────────────────────────┘
```

## 3. 워크플로우별 자동 연계

### 3.1 새 기능 개발 워크플로우

```
사용자: "로그인 기능 추가해줘"
         │
         ▼
┌─────────────────────────────────────────────────────────┐
│  STEP 1: 자동 계획 (planner 에이전트)                    │
│  ┌─────────────────────────────────────────────────────┐│
│  │ 복잡한 기능 감지 → planner 자동 호출                  ││
│  │ 출력: 단계별 계획, 수정 파일 목록, 위험 요소          ││
│  └─────────────────────────────────────────────────────┘│
└──────────────────────────┬──────────────────────────────┘
                           │ 자동 진행
                           ▼
┌─────────────────────────────────────────────────────────┐
│  STEP 2: 테스트 먼저 (tdd-guide 에이전트)                │
│  ┌─────────────────────────────────────────────────────┐│
│  │ TDD 규칙 활성화 → 실패하는 테스트 먼저 작성           ││
│  │ Red → Green → Refactor 사이클                       ││
│  └─────────────────────────────────────────────────────┘│
└──────────────────────────┬──────────────────────────────┘
                           │ UI 작업 감지시
                           ▼
┌─────────────────────────────────────────────────────────┐
│  STEP 3: UI 구현 (frontend-developer 에이전트)          │
│  ┌─────────────────────────────────────────────────────┐│
│  │ React/TypeScript/Tailwind 패턴 자동 적용             ││
│  │ 접근성, 반응형 자동 체크                             ││
│  └─────────────────────────────────────────────────────┘│
└──────────────────────────┬──────────────────────────────┘
                           │ 구현 완료시
                           ▼
┌─────────────────────────────────────────────────────────┐
│  STEP 4: 자동 검증 (/verify 자동 실행)                  │
│  ┌─────────────────────────────────────────────────────┐│
│  │ 1. TypeScript 타입 체크                             ││
│  │ 2. ESLint/Clippy 린트                               ││
│  │ 3. 테스트 실행                                      ││
│  │ 4. 빌드 검증                                        ││
│  │ 실패시 → build-error-resolver 자동 호출              ││
│  └─────────────────────────────────────────────────────┘│
└──────────────────────────┬──────────────────────────────┘
                           │ 검증 통과시
                           ▼
┌─────────────────────────────────────────────────────────┐
│  STEP 5: 보안 검토 (security-reviewer 자동 호출)         │
│  ┌─────────────────────────────────────────────────────┐│
│  │ 새 코드에 대한 보안 취약점 자동 스캔                  ││
│  │ 문제 발견시 → 즉시 알림 및 수정 제안                  ││
│  └─────────────────────────────────────────────────────┘│
└──────────────────────────┬──────────────────────────────┘
                           │ 보안 통과시
                           ▼
┌─────────────────────────────────────────────────────────┐
│  STEP 6: 코드 리뷰 (code-reviewer)                      │
│  ┌─────────────────────────────────────────────────────┐│
│  │ 품질, 성능, 유지보수성 자동 검토                      ││
│  └─────────────────────────────────────────────────────┘│
└──────────────────────────┬──────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  STEP 7: 커밋 준비 안내                                 │
│  ┌─────────────────────────────────────────────────────┐│
│  │ "✅ 모든 검증 통과! 커밋하시겠습니까?"                ││
│  │ → /commit-push-pr 실행 제안                         ││
│  └─────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────┘
```

### 3.2 버그 수정 워크플로우

```
사용자: "이 버그 수정해줘"
         │
         ▼
┌──────────────────────────┐
│ 1. 버그 분석             │
│ 2. 회귀 테스트 먼저 작성  │  ← tdd-guide 자동
│ 3. 수정 구현             │
│ 4. 테스트 통과 확인       │  ← /verify 자동
│ 5. 관련 영향 검토        │  ← code-reviewer 자동
│ 6. 커밋 준비             │
└──────────────────────────┘
```

### 3.3 리팩토링 워크플로우

```
사용자: "이 코드 리팩토링해줘"
         │
         ▼
┌──────────────────────────┐
│ 1. 현재 테스트 확인      │  ← 테스트 없으면 먼저 작성
│ 2. refactor-cleaner     │  ← 데드코드 먼저 제거

└──────────────────────────┘
```

## 5. 에이전트 자동 위임 규칙

### 5.1 트리거 키워드 매핑

| 사용자 요청 패턴 | 자동 위임 에이전트 |
|-----------------|-------------------|
| "계획", "설계", "어떻게 구현" | planner |
| "아키텍처", "구조", "시스템 설계" | architect |
| "UI", "컴포넌트", "화면", "버튼" | frontend-developer |
| "리뷰", "검토", "코드 봐줘" | code-reviewer |
| "보안", "취약점", "안전" | security-reviewer |
| "테스트", "TDD", "커버리지" | tdd-guide |
| "빌드 에러", "컴파일 실패" | build-error-resolver |
| "E2E", "통합 테스트", "Playwright" | e2e-runner |
| "정리", "데드코드", "미사용" | refactor-cleaner |
| "문서", "README", "업데이트" | doc-updater |

### 5.2 자동 위임 플로우

```javascript
// 의사 코드: 자동 위임 로직
function autoDelegate(userMessage) {
  const keywords = extractKeywords(userMessage);

  // 복잡도 분석
  if (isComplexFeature(userMessage)) {
    return delegate('planner');
  }

  // 키워드 기반 위임
  for (const [pattern, agent] of TRIGGER_MAP) {
    if (keywords.match(pattern)) {
      return delegate(agent);
    }
  }

  // 에러 컨텍스트 감지
  if (hasBuildError(context)) {
    return delegate('build-error-resolver');
  }

  // 기본: 직접 처리
  return handleDirectly();
}
```

## 7. 사용자가 알아야 할 것 (최소한)

### 7.1 필수 지식 (3가지만)

```
┌─────────────────────────────────────────────────────────────┐
│  🎯 이것만 기억하세요!                                       │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  1. 작업 요청하면 → 시스템이 알아서 에이전트 호출            │
│                                                             │
│  2. 경고가 뜨면 → 읽고 따르세요                             │
│                                                             │
│  3. 긴 작업 후 → "커밋해줘" 또는 "정리해줘"                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 유용한 명령어 (선택적)

| 상황 | 명령어 | 설명 |
|------|--------|------|
| 복잡한 기능 시작 | `/plan` | 계획 먼저 (보통 자동) |
| 작업 중간 저장 | `/checkpoint` | 상태 저장 |
| 세션 끝내기 전 | `/handoff` | 다음 세션을 위한 인계 |
| 검증 강제 실행 | `/verify` | 린트+테스트+빌드 |
| 커밋하고 싶을 때 | `/commit-push-pr` | Git 통합 |

## 9. 문제 해결 자동화

### 9.1 일반적인 문제 자동 해결

| 문제 | 자동 감지 | 자동 해결 |
|------|----------|----------|
| 빌드 실패 | ✅ | build-error-resolver 호출 |
| 타입 에러 | ✅ | 에러 위치 표시 + 수정 제안 |
| 린트 에러 | ✅ | 자동 수정 가능한 것은 수정 |
| 테스트 실패 | ✅ | 실패 원인 분석 + 수정 제안 |
| 보안 취약점 | ✅ | 즉시 경고 + 수정 방법 제시 |
| 컨텍스트 부족 | ✅ | /handoff 자동 제안 |

### 9.2 에러 복구 체인

```
에러 발생
    │
    ▼
┌─────────────────┐
│ 1차: 자동 수정  │ → 성공 → 계속 진행
└────────┬────────┘
         │ 실패
         ▼
┌─────────────────┐
│ 2차: 에이전트   │ → 성공 → 계속 진행
│    위임        │
└────────┬────────┘
         │ 실패
         ▼
┌─────────────────┐
│ 3차: 사용자     │
│    개입 요청   │
└─────────────────┘
```
